# Melodic PM 포트폴리오 정리

## 1. 프로젝트 한 줄 요약
소규모 공연의 예매, 입금 확인, 모바일 티켓 발급, 체크인, 관객 소통을 **하나의 링크 경험**으로 통합한 공연 운영 플랫폼.

## 2. 문제 정의와 목표
### 문제
- 기존 방식은 **구글폼 예약 → 카카오뱅크 앱 확인 → 수동 매칭 → 링크 전달**로 흐름이 끊겨 있고 운영자 부담이 큼.
- 관객은 **가입/로그인, 티켓 수령, 공연 정보 확인**이 분절되어 경험이 불편함.

### 목표
- 운영자는 **입금 확인/티켓 발급/현장 체크인**을 단일 대시보드에서 처리.
- 관객은 **가입 없이 예약 → 링크 하나로 티켓/정보/소통**을 해결.

## 3. 핵심 가치 제안 (PM 관점)
- **운영 자동화**: 입금 확인과 티켓 발급을 최대한 자동화하여 인력/시간 소모를 줄임.
- **단일 소스 오브 트루스**: 예약/입금/티켓/체크인 상태를 Firestore로 일원화.
- **공연 경험 강화**: 정보 제공 + 응원 게시판으로 공연 전후의 기대감을 유지.

## 4. 스크린 구성과 역할

### Home (`/`)
- 기능: 포스터 ↔ 티켓 플립 카드, QR 티켓 표시, 체크인 완료 배지, 타임라인, 응원 게시판 닉네임 변경.
- 왜 필요한가: 메인 허브에서 **티켓 확인 + 공연 일정 인지**가 동시에 이루어져야 함.
- 업무 도움/차별점: 종이 티켓 없이 QR로 입장 관리, 타임라인으로 현장 문의 감소.

### 관객 입장 (Home 내 Audience Entry)
- 기능: 닉네임 입력 또는 랜덤 생성 후 즉시 입장.
- 왜 필요한가: 공연 응원/정보 접근에서 **가입 장벽 제거**.
- 업무 도움/차별점: “로그인 없이 참여” 경험으로 참여율 상승.

### Reservation (`/reserve`)
- 기능: 이름/연락처/이메일 입력 → 예약 저장 → 계좌 안내 및 입금 가이드(현재 5,000원 기준).
- 왜 필요한가: 예매 의사를 **즉시 예약 데이터로 전환**해야 운영 효율이 높아짐.
- 업무 도움/차별점: 구글폼 없이도 동일 정보 수집, 입금자명 일치 유도.

### 공연 정보 (`/info`)
- 기능: 공연 일시/장소, 1부/2부 셋리스트 제공.
- 왜 필요한가: 공연 전 문의를 줄이고 기대감을 높이기 위한 **디지털 리플렛** 역할.
- 업무 도움/차별점: 공지/안내를 반복 전달하지 않아도 됨.

### 응원 게시판 (`/board`)
- 기능: 포스트잇 UI, 공개/비공개 메시지, 작성/수정/삭제, 공연진 선택.
- 왜 필요한가: 공연 전후의 감정 흐름을 수집하고 소통 채널을 제공.
- 업무 도움/차별점: 단방향 공지에서 벗어나 **양방향 팬 경험** 확보.

### 공연진 로그인 (`/performer/login`)
- 기능: 공연진 로그인/가입, 관리자 기능 접근.
- 왜 필요한가: 운영 영역을 일반 관객과 분리하고 **역할 기반 접근 제어** 필요.
- 업무 도움/차별점: 관리자 기능 보호 및 권한 분리.

### 관리자 대시보드 (`/admin`)
- 기능: 예약 목록 조회, 상태 관리(pending/paid/ambiguous), 승인, 링크 복사, 삭제.
- 왜 필요한가: 입금 확인과 티켓 발급을 **한 화면에서 처리**해야 실수가 줄어듦.
- 업무 도움/차별점: 수기 관리 대신 실시간 리스트로 현황 파악.

### 체크인 데스크 (`/checkin`)
- 기능: QR 스캔 체크인, 수동 토큰 입력, 입장 통계.
- 왜 필요한가: 공연 당일 입장 흐름을 빠르고 정확하게 처리.
- 업무 도움/차별점: 중복 입장 방지, 실시간 입장 수 집계.

## 5. 핵심 사용자 흐름
### 관객 플로우
예약 → 입금 → 티켓 링크(이메일) 수령 → 링크 접속 시 자동 인증 → QR 티켓 표시 → 입장.

### 운영자/스태프 플로우
예약 목록 확인 → 입금 확인(자동/수동) → 티켓 링크 발급 → 체크인 QR 스캔 → 입장 기록.

## 6. 기존 방식 대비 차별점
- **수동 검증 제거**: 구글폼 + 카카오뱅크 앱을 오가며 확인하던 흐름을 단일 시스템으로 통합.
- **발급 자동화**: 상태 변경 시 티켓 링크 자동 발송(이메일)으로 전달 누락 감소.
- **현장 효율화**: QR 체크인으로 입장 관리 시간을 단축.
- **경험 강화**: 공연 정보/응원 게시판으로 “공연 전후 경험”까지 확장.

## 7. 디자이너 관점 요약
- **포스터 → 티켓 플립 카드**: 감성적 경험과 기능(티켓 확인)을 동시에 제공.
- **타임라인 시각화**: 공연 진행 흐름을 직관적으로 전달해 안내 비용 감소.
- **포스트잇 UI**: 응원 메시지에 아날로그 감성을 부여해 참여 심리 강화.

## 8. 개발자 관점 요약
- **Frontend**: React + Vite, React Router, CSS Modules, Framer Motion.
- **Backend/DB**: Firebase Firestore(실시간 동기화), Firebase Auth(공연진 인증).
- **핵심 로직**:
  - `/?auth=TOKEN` 링크로 관객 자동 인증 및 티켓 활성화.
  - Firestore 상태 변경 기반 티켓 발송(Cloud Functions + nodemailer).
  - QR 체크인(@zxing/browser)으로 입장 처리.

## 9. 운영 자동화 설계 (현 상태 + 확장)
- **입금 자동 인입**: MacroDroid 등으로 은행 알림을 읽어 `receiveDeposit` API로 전송 가능.
- **티켓 자동 발송**: 예약 상태가 `paid`로 바뀌면 이메일 발송 트리거.
- **향후 확장**: SMS 발송, 다양한 결제 수단 연동 등.

## 10. 포트폴리오 관점 핵심 성과 포인트
- **운영 효율화**: 한 명 운영자가 전체 예약/입금/티켓/체크인 흐름을 처리 가능.
- **고객 경험 개선**: “로그인 없이 링크 하나로 해결”하는 UX 설계.
- **확장성**: 자동화 파이프라인(입금 알림 → 승인 → 티켓 발송) 구조를 확보.
